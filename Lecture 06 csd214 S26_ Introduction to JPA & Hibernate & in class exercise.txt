1. Introduction: What is JPA and ORM?
The Mismatch (The Impedance Mismatch)
In Lecture 5, you discovered the friction between Object-Oriented Java and Relational Databases:


   * Java uses Inheritance (e.g., Book extends Publication).
   * SQL uses Flat Tables (Rows and Columns).
   * Java uses Associations (Objects pointing to other Objects).
   * SQL uses Foreign Keys and Joins.


An ORM (Object-Relational Mapper) is a library that automates the translation between these two paradigms.
JPA vs. Hibernate
   * JPA (Jakarta Persistence API): This is the Specification. It is a set of interfaces and rules defined by the Jakarta community (the "Java Rulebook"). It includes annotations like @Entity, @Id, and @Column.
   * Hibernate: This is the Implementation. It is the actual, under-the-hood engine that reads your JPA annotations, generates the SQL commands, and handles network traffic to MySQL.
What is an Entity?
An Entity is a lightweight Java class whose state is intended to be persisted in a relational database table [4]. It is a "Java representation of a database row" [4]. Unlike standard POJOs, an Entity is explicitly decorated with metadata annotations (such as @Entity) to describe its relationship to the database schema.

1.1 Deep Dive: JPA Entities


Great Resource:  Java Persistence - Wikibooks, open books for an open world 
	
To understand why enterprise systems depend on JPA, we must analyze the structural requirements of an Entity, the architectural problems it aims to solve, and the practical trade-offs it introduces to software development.
A. Anatomy of a JPA Entity
At its simplest, an Entity is a Java class decorated with metadata. However, the JPA specification enforces strict requirements to ensure that Hibernate can manage the lifecycle of these objects:


   1. @Entity Annotation: Tells the persistence provider (Hibernate) that this class represents a table.
   2. No-Argument Constructor: Must have a public or protected zero-argument constructor [5]. Hibernate uses Java Reflection to instantiate the entity before hydrating its fields. If omitted, the application will fail on startup.
   3. @Id Annotation: Every entity must have a designated primary key. This is used by JPA to uniquely identify the row in the persistence context.
   4. Non-Final Class & Fields: The entity class and its persistent fields must not be declared final. Hibernate subclass-proxies entities to support features like Lazy Loading (loading children only when accessed); final keywords prevent this subclassing.


________________


B. The "Why": Solving the Impedance Mismatch
Historically, developers wrote raw SQL strings directly inside Java files. This created a friction point known as the Object-Relational Impedance Mismatch.


Java objects are structured as graphs with inheritance, polymorphic relationships, and dynamic behaviors. Relational databases are structured as flat, rigid grids of keys, rows, and constraints. JPA entities act as the declarative bridge, allowing developers to define how these two conflicting paradigms should align without writing manual translator scripts for every table.


________________


C. The Pros (Advantages of JPA Entities)
   1. Elimination of Boilerplate DTO/JDBC Code:
In raw JDBC, inserting a nested object hierarchy requires writing multiple SQL statements, managing transactional savepoints, and mapping auto-incremented keys manually. JPA automates this entire process. A single call to em.persist(entity) generates the required SQL under the hood.
   2. Database Portability (Dialect Abstraction):
JPA entities are database-agnostic. The SQL dialect is configured in persistence.xml rather than hardcoded in the application. If an organization migrates its database from MySQL to PostgreSQL, developers only need to change the hibernate.dialect property. Hibernate will automatically translate the entity queries into PostgreSQL-compliant SQL syntax.
   3. Automated Lifecycle Tracking (Dirty Checking):
Inside a transaction, JPA places entities in a Managed state. Hibernate takes a snapshot of the object's fields. When the transaction commits, Hibernate compares the current state of the object against the snapshot. Any modifications automatically trigger the appropriate UPDATE statements without requiring the developer to call an explicit "update" method.
   4. Caching and Performance Optimizations (First-Level Cache):
The EntityManager maintains a First-Level Cache (Persistence Context) within a single transaction. If your application queries the same book entity three times in a single request, JPA hits the database once, stores the result in memory, and returns the cached object for subsequent requests. This reduces redundant network overhead.


________________


D. The Cons (Disadvantages & Trade-offs of JPA Entities)
      1. Steep Learning Curve and "Hidden" Complexity:
While basic CRUD operations are straightforward, mapping complex relationships (@OneToMany, @ManyToMany) requires a deep understanding of JPA mechanics. Mistuning fetch strategies can lead to performance catastrophes, most notably the N+1 Query Problem (where a query designed to fetch 100 records mistakenly executes 101 separate database hits).
      2. Memory and CPU Overhead:
Because Hibernate must perform snapshotting, dirty-checking, reflection-based instantiation, and maintain the First-Level Cache, JPA consumes substantially more JVM heap memory and CPU cycles than raw JDBC or lightweight row mappers (like Spring's JdbcTemplate).
      3. Loss of Precise SQL Control:
Since Hibernate generates the SQL statements automatically, developers lose granular control over the query structure. For highly complex queries involving complex analytic functions, CTEs (Common Table Expressions), or database-specific optimizations, the generated JPQL can be highly inefficient compared to a hand-tuned native SQL query.
      4. "Magic" is Hard to Debug:
When a database transaction fails due to constraint violations or synchronization issues, debugging can be difficult. Developers must look past the Java stack trace, enable SQL logging (hibernate.show_sql=true), and analyze the raw SQL output to determine what the ORM engine did incorrectly.

Analyzing hibernate.hbm2ddl.auto options:
         * create: Drops existing tables and creates fresh ones on application startup [5]. (Best for development and testing) [5].

         * none: Prevents Hibernate from making schema changes. (The standard for Production environments).

4. Simple Standalone JPA: JpaBookApp.java
Open src/main/java/bookstore/jpa/JpaBookApp.java. This script is a standalone demonstration of the complete JPA transaction lifecycle.


package bookstore.jpa;


import bookstore.entities.BookEntity;
import jakarta.persistence.EntityManager;
import jakarta.persistence.EntityManagerFactory;
import jakarta.persistence.Persistence;
import java.util.List;


public class JpaBookApp {
    public static void main(String[] args) {
        // 1. Start Hibernate (Loads persistence.xml configuration)
        EntityManagerFactory emf = Persistence.createEntityManagerFactory("bookstore-pu");
        EntityManager em = emf.createEntityManager(); // Opens a session to the DB
        
        try {
            // 2. Create Data inside a Transaction
            em.getTransaction().begin();
            
            BookEntity book = new BookEntity();
            book.setPrice(34.99);
            book.setCopies(10);
            book.setAuthor("Maria Batista");
            
            em.persist(book); // Hibernate generates the INSERT SQL statement automatically
            em.getTransaction().commit(); // Permanently saves changes to MySQL
            System.out.println("Book persisted with ID: " + book.getId());


            // 3. Read Data (Polymorphic Query)
            // We run JPQL (Java Persistence Query Language) - Querying the Class, not the table
            List<BookEntity> books = em.createQuery(
                    "SELECT b FROM BookEntity b", BookEntity.class).getResultList();


            for (BookEntity b : books) {
                System.out.println("Found persistent entity: " + b);
            }
        } finally {
            // 4. Safely Close database connections
            em.close();
            emf.close();
        }
    }
}

5. Architectural Critique: Why Not Extend Editable in Entities?
In Labs 1 and 2, our data classes extended Editable to directly handle CLI Scanner input. Why can't we just annotate those classes as @Entity and save time?


This is a critical architectural decision based on three clean-code principles:


         1. Single Responsibility Principle (SRP):
An Entity has one job: representing database state [6]. Editable has one job: talking to a human via the command line [6]. If we merge them, our persistence code becomes tightly coupled to a CLI [6]. If we migrate to a Web Server later, the Scanner code inside our entities will cause the application to crash [6].
         2. The DTO (Data Transfer Object) Pattern:
POJOs (like Book.java and Tire.java) act as DTOs [6]. They are temporary containers that collect and validate user keyboard inputs [6]. Once data is clean, we map (copy) it into the Entity (the "Vault") for database storage [6].
         3. The "Long Transaction" Problem:
Database connections are scarce. JPA transactions should be extremely short: Open → Write → Close [6]. If we put Scanner prompts inside an Entity transaction, the database connection stays locked while the user is typing [6]. If the user goes on break, the connection is wasted [6].

       return price;

   }

       this.manufacturer = manufacturer;
       this.price = price;

       return Objects.hash(manufacturer, price);

               ", price=" + price +

Create a new standalone class src/main/java/bookstore/jpa/JpaAutomotiveApp.java to test your automated database CRUD lifecycle:




package bookstore.jpa;


import bookstore.entities.TireEntity;
import jakarta.persistence.EntityManager;
import jakarta.persistence.EntityManagerFactory;
import jakarta.persistence.Persistence;
import java.util.List;


public class JpaAutomotiveApp {
    public static void main(String[] args) {
        EntityManagerFactory emf = Persistence.createEntityManagerFactory("bookstore-pu");
        EntityManager em = emf.createEntityManager();


        try {
            // --- CREATE ---
            System.out.println("\n[Step 1] Persisting a new Tire...");
            em.getTransaction().begin();
            TireEntity myTire = new TireEntity("Michelin", 249.99, 18);
            em.persist(myTire); // Hibernate generates INSERT
            em.getTransaction().commit();
            System.out.println("Tire saved with Database Primary Key ID: " + myTire.getId());


            // --- READ ---
            listTires(em, "[Step 2] Database Tires:");


            // --- UPDATE (Dirty Checking / Managed Entities) ---
            System.out.println("\n[Step 3] Editing Tire Price (Dirty Checking)...");
            em.getTransaction().begin();
            TireEntity managedTire = em.find(TireEntity.class, myTire.getId());
            if (managedTire != null) {
                managedTire.setPrice(199.99); // Changing Java field updates DB on commit
            }
            em.getTransaction().commit();
            listTires(em, "[Step 4] After Price Update:");


            // --- DELETE ---
            System.out.println("\n[Step 5] Deleting the Tire...");
            em.getTransaction().begin();
            TireEntity target = em.find(TireEntity.class, myTire.getId());
            if (target != null) {
                em.remove(target); // Hibernate generates DELETE
            }
            em.getTransaction().commit();
            listTires(em, "[Step 6] After Deletion:");


        } finally {
            em.close();
            emf.close();
        }
    }


    private static void listTires(EntityManager em, String header) {
        System.out.println("\n" + header);
        List<TireEntity> tires = em.createQuery("SELECT t FROM TireEntity t", TireEntity.class).getResultList();
        if (tires.isEmpty()) {
            System.out.println("No tires found.");
        } else {
            tires.forEach(t -> System.out.println(" > " + t.getManufacturer() + " ($" + t.getPrice() + ")"));
        }
    }
}

7.1 Architectural Evolution: Introducing JPA Persistence, the  DTO pattern  & Separation of Concerns (SRP)
Up to this point, our bookstore application managed its runtime state inside a volatile, in-memory ArrayList<SaleableItem> located within App.java. While this collection-based approach served as a straightforward way to learn object-oriented design, it possessed a critical limitation: all inventory and transaction data vanished the moment the Java Virtual Machine (JVM) was shut down.


To resolve this limitation and provide true data persistence, we are transitioning our data management engine to the database. We will replace the volatile ArrayList in our core controller (App.java) with a persistent JPA EntityManager [5].


However, moving to a database-backed persistence model raises an important architectural question: Where should our user interaction logic reside?


In earlier labs, our business classes extended Editable to directly handle console scanning and terminal inputs. If we simply decorate those same classes with JPA annotations (such as @Entity) to save them to database tables, we create a severe architectural conflict. Database model classes (Entities) should remain clean, lightweight representations of our database schema, completely isolated from user-interface technologies [4, 6].


To solve this problem without violating the Single Responsibility Principle (SRP), we are introducing the Data Transfer Object (DTO) design pattern [6]. This allows us to separate our user-interface presentation layer from our persistent database schemas, establishing a clean boundaries-and-layers design that makes our system simpler to test, scale, and maintain.


What is a DTO (Data Transfer Object)?
A Data Transfer Object (DTO) is an object designed to carry data between software processes or architectural layers without containing any business or persistence logic.


In our refactored bookstore application, our original POJOs (Book, Ticket, Tire, Battery) have been repurposed to act as DTOs. They exist solely to capture input from the user console via the Editable interface and carry that structured data across the boundary to the persistence layer.
Why Do We Use Them? (The SRP Justification)
The Single Responsibility Principle (SRP) dictates that a class should have only one reason to change.


In a basic, non-DTO architecture, you might be tempted to annotate your Product classes directly as database @Entity classes while keeping the Editable console scanning logic. Doing so introduces several severe architectural flaws:


               1. Tight Coupling to Presentation: If an Entity is directly responsible for database mappings and command-line prompts (Scanner), your database layer is locked into a console application. If the bookstore decides to build a web interface using HTML forms later, you cannot reuse your entities without bringing along dead Scanner code.
               2. Leaking Database Auto-Generated Keys: Database primary keys (@Id Long id) are an internal implementation detail of your database schema. They should not be exposed to the user interface. DTOs isolate these database-specific columns from the presentation layer.
               3. Protecting Database Model Integrity: Users should never have direct, unvalidated write access to database entities. DTOs act as a buffer. The user manipulates the DTO, the application validates the properties, and only then is the validated data mapped into the Entity class and persisted.

7.2 Architectural Overview of the App Refactoring
To support this decoupled architecture, the data flow of our application has changed fundamentally. Instead of holding temporary references inside an in-memory ArrayList<SaleableItem>, our controller (App.java) coordinates database state changes by mapping DTOs to Entities inside active JPA transactions.
The Lifecycle of a Write Operation (Create)
[User Input] ──> [DTO POJO (e.g., Book)] ──> [toEntity()] ──> [BookEntity] ──> [JPA EntityManager] ──> [MySQL Disk] 


               1. The user selects "Add Book."
               2. An empty DTO (Book POJO) is instantiated.
               3. The DTO calls initialize(Scanner) to capture keyboard inputs.
               4. The validated DTO calls .toEntity(), producing a BookEntity.
               5. App.java opens a database transaction and persists the BookEntity via the EntityManager.
The Lifecycle of a Read Operation (List)

[MySQL Disk] ──> [JPA EntityManager] ──> [BookEntity] ──> [fromEntity()] ──> [Book DTO] ──> [Console Output] 


               1. The user selects "List Books."
               2. App.java queries the database for all persistent entities using JPQL: SELECT b FROM BookEntity b.
               3. The EntityManager returns a list of managed BookEntity objects.
               4. App.java translates these database rows back into safe, presentation-ready DTOs using Book.fromEntity().
               5. The DTO's toString() method is output to the user's console screen.

                        faker.number().numberBetween(50, 200),
                        faker.date().past(60, TimeUnit.DAYS),
                        faker.bool().bool()
                );
                em.persist(dm);


                // 4. Seed Ticket

                t.setDescription("Concert: " + faker.rockBand().name());

                        faker.company().name() + " Tires",        // manufacturer
                        faker.number().randomDouble(2, 80, 350),  // price
                        faker.number().numberBetween(15, 22)      // diameter
                );

        }
    }
